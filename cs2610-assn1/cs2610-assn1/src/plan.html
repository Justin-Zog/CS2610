  3. Handle Errors and alternate filepath names
  4. The server will take in input from the user in the form of a 'filepath' request

Phase 1: Design

Deliver

I want a function that handles all of my filetypes no matter the kind.
It will take the file the user input and decide what to do with it.
If the file is an alternate name then the function will use a 301 request and perform another do_GET request with the correct pathname
If the file is recognized it will do a 200 and send the correct headers, etc.
An unrecognized path will take the user to the 404 page.

There will be another function that takes care of sending the correct headers for the type of response (200, 301, etc.)

Phase 3: Testing & Debugging

Deliver

I kept getting a broken pipe error. After doing some research I determined that this was because I was writing the Connection: close header before some of the others. Once I fixed that and implemented the end_headers this error was mostly solved... Except for at random times. This confused me but Professor Falor helped clear my concerns.

I kept getting a weird error where the server would send the headers two times instead of one and display the second sent headers on my debugging page only. This was solved by putting a return in the function so the code at the bottom didn't get hit (sending the second sendHeaders)

I also ran into an odd problem in only safari where if I typed a bogus url in http://localhost:8000/fluffy minutia Safari would automagically perform a search for the entire url instead of taking me to the 404 page. I never solved this error. I think it just a quirk of Safari and typing in the above url on other browsers took me to the 404 page.

I tested all the alternate links and inspected the network tab to make sure my server was sending the correct codes.
